# Linux: базовая теория

## 1. Базовые команды

| Команда     | Что делает                                   |
|------------|----------------------------------------------|
| `cd`       | Перейти в директорию.                        |
| `mkdir`    | Создать директорию.                          |
| `rm`       | Удалить файл.                                |
| `rm -r`    | Удалить директорию рекурсивно.               |
| `rm -rf`   | Жёстко удалить директорию со всем содержимым (без вопросов, опасно). |
| `nano` / `vim` | Редактировать файл в терминале.          |
| `touch`    | Создать пустой файл или обновить время изменения. |
| `chmod`    | Изменить права доступа файла/директории.     |

---

## 2. Редакторы

- `nano` — простой, с подсказками внизу, подходит для быстрых правок.  
- `vim` — модальный редактор, требует привычки, но даёт очень быстрый набор и редактирование при регулярном использовании.

---

## 3. Куда углубляться

Здесь вещи, которые превращают набор команд в удобный инструмент админа: поиск, пайпы, текстовые фильтры, простые скрипты и SSH с ключами.

### 3.1 Навигация и работа с файлами

#### find — поиск файлов по условиям

`find` ходит по дереву каталогов и выбирает файлы по имени, типу, размеру, дате.

```bash
# все .log в /var/log
find /var/log -type f -name "*.log"

# всё, что больше 100 МБ в /var
find /var -type f -size +100M

# найти и удалить *.tmp под текущей директорией
find . -type f -name "*.tmp" -delete
```

Сценарии: поиск тяжёлых файлов, мусора, конфигов по маске.

#### grep — поиск по содержимому

`grep` ищет строки по шаблону в файлах или в потоке.

```bash
# все ERROR в системном логе
grep "ERROR" /var/log/syslog

# «failed password» в auth.log, без учёта регистра, с номерами строк
grep -ni "failed password" /var/log/auth.log

# поиск строки во всех файлах каталога
grep -R "my-service" /etc
```

Это основной способ быстро вытащить из логов нужные события.

---

### 3.2 Потоки, перенаправления, пайпы

Работаем с тремя потоками: stdin, stdout, stderr. Главное — уметь направлять их как надо.

#### Перенаправления

```bash
# записать вывод в файл
ls /etc > etc-list.txt

# дописать в конец файла
ls /var >> etc-list.txt

# ошибки в отдельный файл
my_script.sh 2> errors.log

# и вывод, и ошибки в один лог
my_script.sh > script.log 2>&1
```

Так логируют скрипты и разделяют обычный вывод и ошибки.

#### Пайпы `|`

Пайп передаёт stdout одной команды на stdin другой, получаются цепочки обработки.

```bash
# последние 50 строк с ERROR
grep "ERROR" /var/log/syslog | tail -50

# сколько раз IP светился в access.log
grep "1.2.3.4" /var/log/nginx/access.log | wc -l
```

Паттерн: «команда, которая много печатает» → фильтр → счётчик/обрезка.

---

### 3.3 Текстовые фильтры: sort, uniq, wc, xargs

#### sort, uniq, wc — подсчёт и статистика

```bash
# топ-10 IP по количеству запросов в access.log
awk '{print $1}' /var/log/nginx/access.log \
  | sort \
  | uniq -c \
  | sort -nr \
  | head -10
```

Что происходит:

- `awk '{print $1}'` — оставляем только колонку с IP.  
- `sort` — сортируем.  
- `uniq -c` — склеиваем одинаковые строки и считаем количество.  
- `sort -nr` — сортируем по количеству по убыванию.  
- `head -10` — берём топ‑10.

`wc` — быстрый способ оценить объём:

```bash
# количество строк (обычно = количество записей в логе)
wc -l /var/log/syslog
```

#### xargs — мост между командами

`xargs` берёт строки из stdin и превращает их в аргументы команды.

```bash
# удалить все *.tmp, найденные find (аккуратно с пробелами в именах)
find . -type f -name "*.tmp" -print0 | xargs -0 rm -f
```

### 3.4 Bash‑скрипты

Скрипт — файл с командами, которые ты и так вводишь руками, плюс немного логики.

#### Базовый скелет скрипта

```bash
#!/bin/bash
set -euo pipefail

LOG=/var/log/syslog

if grep -q "ERROR" "$LOG"; then
  echo "[$(date)] Есть ошибки в $LOG"
else
  echo "[$(date)] Ошибок не обнаружено"
fi
```

Ключевые моменты:

- `#!/bin/bash` — шебанг.  
- `set -euo pipefail` — не игнорировать ошибки и косяки в пайпах.  
- переменные и кавычки вокруг путей.

Запуск:

```bash
chmod +x check_errors.sh
./check_errors.sh
```

#### Цикл по файлам

```bash
#!/bin/bash
set -euo pipefail

for file in /var/log/nginx/*.log; do
  lines=$(wc -l < "$file")
  echo "$(date) $file: $lines строк"
done
```

Паттерн: пробежать по набору файлов, для каждого сделать действие (сжать, скопировать, собрать статистику).

### 3.5 SSH, ~/.ssh/config, scp/rsync

#### Ключи и вход без пароля

```bash
# создать пару ключей
ssh-keygen -t ed25519 -C "devops-key"

# закинуть публичный ключ на сервер
ssh-copy-id user@server
```

После этого:

```bash
ssh user@server
```

заходит без пароля (если на сервере включены логины по ключам).

#### ~/.ssh/config — алиасы для серверов

```bash
Host dev-vpn
  HostName 1.2.3.4
  User root
  Port 2222
  IdentityFile ~/.ssh/id_ed25519
```

Теперь:

```bash
ssh dev-vpn
scp config.yaml dev-vpn:/etc/myapp/
```

Это сокращает команды и уменьшает шанс ошибиться в адресе/порте.

#### Копирование: scp и rsync

Разовая отправка файла:

```bash
scp backup.tar.gz dev-vpn:/root/backups/
```

Синхронизация директорий (деплой, бэкапы):

```bash
rsync -avz ./app/ dev-vpn:/srv/app/
```

- `-a` — сохранить права/владельцев/время.  
- `-v` — подробный вывод.  
- `-z` — сжатие по сети.

## 4. Процессы

Процесс — запущенная программа с собственным PID и ресурсами.

| Команда / понятие | Что делает / что это                         |
|-------------------|----------------------------------------------|
| `ps aux`          | Показать все процессы в системе.            |
| `ps -p PID -o ...`| Детальная инфа по одному PID.               |
| PID               | Уникальный идентификатор процесса.          |
| `kill <PID>`      | Отправить сигнал процессу (обычно SIGTERM). |
| `kill -9 <PID>`   | SIGKILL, жёстко убить процесс.              |

Частые сценарии: найти «жрущий» процесс, посмотреть, чем занят сервис, аккуратно его остановить/перезапустить.

---

## 5. Директория `/proc`

- Псевдо‑файловая система, отражающая состояние ядра и процессов.  
- `ps`, `top`, `htop` читают оттуда инфу.  
- В `/proc/<PID>/` лежит информация о конкретном процессе: командная строка, открытые файлы, лимиты и т.д.

Полезно для диагностики: что именно запущено, какие дескрипторы открыты, какие лимиты стоят.

---

## 6. Systemd, юниты и OOM Killer

- `systemd` — процесс с PID 1, который запускает остальные сервисы и следит за ними.  
- Юнит‑файлы (`*.service`, `*.timer` и т.д.) описывают, как стартовать сервис, под каким пользователем, как перезапускать.  
- Управление:  
  - `systemctl start/stop/restart/status <name>.service`  
  - `systemctl enable/disable <name>.service`  
  - `journalctl -u <name>.service` — логи юнита.  

OOM Killer:

- Когда системе не хватает RAM, ядро выбирает «жертву» и убивает процесс, чтобы освободить память.  
- В логах это видно как сообщения про `Out of memory` и `Killed process ...`.  
- Важно понимать, что может упасть не только самый «жирный» процесс, но и тот, у которого ниже приоритет по OOM.

# Git: базовая теория

## Зачем нужен Git

Git — это распределённая система контроля версий: она хранит историю изменений файлов, позволяет работать в ветках и безопасно объединять работу разных людей в одном проекте [web:3].  
Каждое сохранение состояния проекта в Git называется **коммитом**, а вся история коммитов образует направленный граф снимков проекта [web:6].

## Ключевые концепции

### Ветки (branches)

Ветка — это «линия разработки», указатель на цепочку коммитов.  
Обычно есть основная ветка (`main` или `master`), куда попадает стабильный код, и дополнительные ветки для фич, экспериментов и фиксов [web:3].

Основные идеи работы с ветками:

- Создаём отдельную ветку под задачу (feature, bugfix).  
- Периодически обновляем её из основной ветки через `merge` или `rebase` [web:8].  
- После завершения работы вливаем изменения обратно в основную ветку.

### Теги (tags)

Тег — это «ярлык» на конкретный коммит, чаще всего используется для пометки версий релизов (например, `v1.0`, `v2.0`) [web:3].  
Теги помогают быстро вернуться к состоянию проекта на момент определённого релиза.

Примеры (для справки):

- Лёгкий тег: `git tag v1.0`  
- Аннотированный тег с описанием: `git tag -a v1.0 -m "first release"` [web:3].

### Markdown (.md)

Markdown — простой текстовый формат разметки, который хорошо подходит для README, документации и конспектов [web:3].  
Файлы с расширением `.md` удобно просматривать на GitHub/GitLab: заголовки, списки, ссылки и код подсвечиваются автоматически.

## 5 базовых команд Git

Эти команды покрывают типичный цикл работы «инициализация → изменение → сохранение → отправка → получение» [web:2][web:6].

| Команда   | Что делает |
|----------|-----------|
| `git init`   | Инициализирует новый локальный репозиторий в текущей директории [web:3]. |
| `git add`    | Добавляет (подготавливает) файлы к следующему коммиту, помещая их в staging area [web:6]. |
| `git commit` | Фиксирует подготовленные изменения в истории с сообщением коммита [web:6]. |
| `git push`   | Отправляет локальные коммиты в удалённый репозиторий (например, на GitHub) [web:6]. |
| `git pull`   | Забирает изменения из удалённого репозитория и сливает их с текущей веткой [web:9]. |

Мини-пример цикла работы:

1. Изменяем файлы в проекте.  
2. `git add .` — подготовка всех изменённых файлов [web:6].  
3. `git commit -m "Краткое описание изменений"` — создание коммита [web:6].  
4. `git push origin main` — отправка в удалённый репозиторий [web:6].

## Что ещё стоит выучить

### Дополнительные команды

- `git checkout` — переключение между ветками или возврат к конкретному коммиту [web:8].  
- `git merge` — слияние одной ветки в другую, сохраняет обе истории и создаёт merge-коммит [web:8].  
- `git rebase` — «перепроигрывает» коммиты ветки поверх другой базы, делая историю линейной [web:8][web:11].  
- `git fetch` — скачивает новые коммиты и ветки из удалённого репозитория без автоматического слияния [web:8].  
- `git diff` — показывает различия между версиями файлов (рабочая директория, индекс, коммиты) [web:3].  
- `git clone` — создаёт локальную копию (реплику) существующего удалённого репозитория [web:3].

### Стратегии ветвления

**Стратегия ветвления** — это набор правил, как команда создаёт ветки, когда их сливает и как выпускает релизы [web:7].  

Популярные варианты:

- GitFlow — выделенные ветки под разработку, релизы и хотфиксы, подойдёт для сложных продуктов с релизными циклами [web:7].  
- GitHub Flow — простая модель: ветки фич → pull request → merge в `main`, часто используется для веб‑сервисов с частыми деплоями [web:7].  
- Trunk-Based Development — почти всё делается в одной «общей» ветке (trunk), мелкие изменения вливаются очень часто, что ускоряет CI/CD [web:7].

## Мини-шпаргалка по рабочему процессу

Типичный сценарий для учебного или пет-проекта:

1. Создать или клонировать репозиторий.  
2. Работать в отдельной ветке под задачу.  
3. Делать небольшие осмысленные коммиты с понятными сообщениями.  
4. Регулярно забирать изменения с удалённого репозитория (`git fetch`, `git pull`) [web:8][web:9].  
5. Перед слиянием решать конфликты и следить, чтобы основная ветка оставалась в рабочем состоянии [web:8].
